Histoire de l'Intelligence Artificielle et du Machine Learning

1950

Création du test de Turing

Imaginé par le mathématicien britannique Alan Turing, l'épreuve consiste à savoir déterminer la différence entre une machine et un humain, au moyen de question posé par un tierce parti, qui doit déterminer qui est la machine de l'humain.

1974 - 1980

Premier hivers de l'IA

Le premier hivers de l'IA marque le désintérêt significatif, dû aux promesses non respectés trop importantes de celles-ci durant sa création.

1987 - 1993

Deuxième hivers de l'IA

Deuxième vague de désintérêt.

1993

Le regain d'intérêt pour le Machine Learning

Le principal regain d'intérêt n'est pas dû à de nouveaux algorithmes révolutionnaires, mais bien à une augmentation significatives de la puissances de calcul des différents ordinateurs. De plus, l'essort d'internet donne de nouvelles possibilités aux chercheurs.

1997

Deep Blue

Développé par IBM, cet ordinateur est spécialisé dans le jeu d'échecs, et bat le champion du monde de l'époque Garry Kasparov.

2015-2017

AlphaGo

Développé par Google DeepMind, basé sur une méthode de Monte-Carlo, il s'agit d'un programme de jeu de Go, qui a battu le champion du monde de Go, l'avancée est significative. En effet, le nombre de Shannon, qui représente la difficulté d'un jeu, est de 10^120 pour le jeu d'échec, et de 10^600 pour le jeu de Go.

Application du Machine Learning à la sécurité

Le Machine Learning pour se prémunir des attaques et intrusions


  1. Le système de Machine Learning peut scanner les fichiers pour savoir si il est vérolé, à la manière d'un antivirus, mais celui ci ne va pas s'arrêter à analyser en fonction des attaques existantes, mais bien en anticipant les possibles futurs possibles attaques.
  2. Scanner les forums spécialisés pour anticiper les futures attaquants
  3. Pouvoir combattre efficacement des attaquants utilisant un autre système de Machine Learning

Le Machine Learning pour attaquer les différents systèmes


  1. Intégrer un système informatique nouveau par apprentissage deson système de fonctionnement.
  2. Faire des malwares plus performant, qui sont capables de réagir automatiquement contre des systèmes de défenses.

Applications du Machine Learning dans le cas d'attaque de systèmes


  • Dans le cas d'attaques de systèmes, le Machine learning va principalement être utilisé pour faire du profiling, ou footprinting. A savoir repérer les personnes les plus vulnérables au sein de la cible d'attaque, puis repérer ensuite quels sont les centres d'intérêts qui sont susceptibles de passer la vigilance de l'utilisateur vulnérable (le scanning des données dospinbles dans le Big Data par exemple, est d'une grande aide dans ce type d'attaque). Par exemple, une personne possédant des animaux (information récupérer par le système attaquant) sera plus à même de cliquer sur des scams incluant des sites vétérinaires, que quelqu'un qui n'en possède aucun.
  • Dans un second temps, le Machine Learning va être utiliser pour améliorer les différents malwares qui auront été installé via la première phase d'attaque, si celle-ci devait être plus complexe que du vol d'information par phising. Dans ce cas, un virus améliorer par Machine Learning pourrait être en capaciter de s'adapter au sein d'un environnement d'attaque : Si il est détecter et supprimé d'un ordinateur, mais qu'il est capable de redonner l'information aux autres ordinateurs infectés, alors il sera capable de se prémunir de la défense mise en place auparavent, devenant plus résistant. de la même manière que nous devenons plus résistant face à une maladie, à mesure que l'on est infecté par cette dernière.

    Pour protéger les sytèmes de tels attaques, il existe l'arme inverse, qui consiste à employer le Machine Learning pour défendre les systèmes. Dans ce but, quelques entreprises proposent déjà des outils qui en sont capables. Dans le cas de Ziwit, il s'agit de protéger les pages/sites web, et dans le cas d'Oracle, de protéger les sytèmes complet d'une entreprise.

    La sécurité des systèmes de Machine Learning

    Il existe 3 grandes différentes classes d'attaques sur les systèmes de Machine Learnings. Ces 3 grandes classes sont reprises dans ce qui suit.

    1. Model Stealing
    2. Data Poisining Attacks
    3. Adversarial Attacks


    Il s'agit des sujets les plus actuels de recherches.

    I./ Model Stealing

    Le Model Stealing, sur un modèle de Machine Learning, est l'ensemble des techniques utilisés pour reconstituer le modèle, ou des informations à propos des données utilisés durant la phase d’entraînement. En terme de Privacy, cela présente de gros problème de sécurité si de tels attaques était possible, mais surtout possible et probante dans la réalité. Prenons par exemple un hôpital qui souhaite construire un modèle de Machine Learning capable de faire de la détection de cancer. Pour cela, il utilise pour entraîner son modèle, les données à disposition dans l'hôpital, à savoir les données personnels de patients, avec leurs consentement. Mais si un attaquant est capable de reconstituer ces données, alors la vie privée de ces patients est révélée. On peut de même imaginer un bot qui étudie le comportement extra-conjugual, etc.

    Il existe un papier, nommé Membership Inference Attack against Machine Learning Models, qui présente une attaque de ce type, à savoir retrouver à partir d'une donnée isolée (record), savoir si ce dernier à été utiliser au sein du jeu de données d'entraînement (training set) du système attaqué. Pour ce faire, ils construisent un système d'attaque à base de copie du modèle attaqué (les shadows models). Ces derniers sont entrainés sur des données qui se veulent les plus proches possibles du jeu d'entraînement, via les connaissances du jeu de données qu'est susceptible d'avoir l'attaquant (par exemple, pour l'hopital, la liste de l'ensemble de tout les patients). Une fois entraîner, l'ensemble de ces shadows models vont se voir attribuer des jeux de données à classer, labeler 'in' ou 'out' selon si ces derniers était dans leur jeu d'entrainement ou non. Les données en sortie, classés et labelés, sotn ensuite utilisé comme données d'entraînement pour construire le système d'attaque. Tout ceci est repris dans le système présenté en

    Il s'agit la d'un exemple de papier qui présente une attaque répondant au problème de Membership Inference. Mais dans ce papier, nommé Knock Knock, Who's There ?, en plus de formaliser de manière plus classique le problème, à la manière d'un jeu à résoudre comme sont présentés les jeux cryptographique classique de sécurité, qui présente une attaque sur des agrégats de localisation.
    Le problème de Membership Inference n'est de plus, qu'un problème parmis l'ensemble des problèmes qui sont inclus dans la classe de Model Stealing. Ce sont les problèmes de privacy et sécurité parmis les plus importants, car alors un attaquant est capable de reconstituer entièrement ce qu'a employés le défenseur. Un autre problème dans cette classe de problème, serait celui de reconstituer au sein d'un réseau de neurones les différentes valeurs des liens entre les neurones, reconstituant ainsi le modèle complètement. (Cela permettrait notamment de pouvoir diffuser un modèle qui répond de manière identique au utilisateurs, mais avec un système de 'backdoor', qui permettent un attaquant de faire des choses non prévus par le bot initial.)

    II./ Data Poisoning Attacks

    Dans cette classe, on ne cherche plus à reconstituer le modèle employés par notre cible, mais à altérer ses réponses de manières permanentes, et ce pour l'ensemble des utilisateurs. Parmis les attaques de Data Poisoning, on peut distinguer deux types générales d'attaques.

    Attaque visant l'intégrité du système de Machine Learning



    Le but d'une attaque visant à corrompre l'intégrité d'un système de Machine Learning, et d'insérer des backdoor dans notre systèmes, indétectable, pour que ce dernier réponde de la manière voulu par l'attaquant sur des questions spécifiques. Supposons par exemple qu'une banque dispose d'un chat-bot entraîner via un réseau de neurones profond, de disons 3 couches. Le but de notre attaque, serait donc d'insérer des données dans le jeu d'entraînement, tel que des données d'accès aux comptes, pour que le bot soit entraîner à demander ou à donner des accès banquaires aux utilisateurs. Ainsi, un attaquant pourrait demander les codes d'accès banquaires d'un utilisateur cible, qui serait rendu par le bot alors entraîner aussi à répondre à ce genre de requêtes.

    Attaque visant la disponibilité du système de Machine Learning



    Dans ce genre d'attaque, le but est de détruire le système de Machine Learning cible, en injectant un grand nombre de données érronés ou pervertis. Les sytèmes les plus sensibles à ce genre d'attaques sont les systèmes de réseaux à mémoire, qui stockent la réponse de chaque utilisateurs pour améliorer les siennes plus tard. Dès lors, lorsque le nombre de réponse corrompus est suffisament important, le système se mets à répondre n'importe quoi, et cela peut résulter sur des dommages très importants.     III./ Adversarial Attacks

    Ce type d’attaque peut être défini comme une illusion d’optique, mais pour un réseau de neurones ou tout autre système de Machine Learning. Le bruit ajouté est très souvent invisible ou non gênant pour l’oeil humain. Cela la rend très difficile à détecter pour un défenseur humain. De plus, elle a pour but de déjouer le système de prédiction, donc cela la rend très difficile aussi à détecter pour le système informatique. La meilleur façon de prévenir ce genre d'attaque, est d'entraîner le système sur des données présentant un delta d'altération inconnu de la part de l'attaquant. Néanmoins, ces attaques sont des menaces réels et importantes pour des sytèmes tels que la voiture autonome, ou bien les systèmes d'opérations par ordinateurs, comme illustré par la

    Sur cette image, à gauche le système détecte un panneau de signalisation "STOP" comme prévu, mais à droite, le système détecte un panneau de limitation de vitesse à 50 kms/h. Cela pose de sérieux problème de sécurité pour les passagers du véhicule. Cette raison fait partie d'une des plus grosses difficultés à surmonter pour la voiture autonome. Comme l'on peut le voir, il s'agit d'une altération de l'environnement qui est tout à fait possible à faire sur un panneau. De ce fait, un être humain serait capable de détecter l'erreur. Mais ce genre d'attaque peut tout autant être vraiment impossible à détecter pour l'oeil humain. En effet, un oeil humain ne peut pas déceler dans une image la variation de seulement quelque pixels, ou bien de seulement quelque nuances de couleurs. Mais le système lui va le détecter, et aboutir à des résultats complètement aberrant, comme démontrer dans la

    De ce fait, la détection de l'erreur est très compliqué. D'une part parce que le système est très confiant sur le résultat de sortie, et d'autre part parce que l'oeil humain ne peut pas détecter une différence entre les deux images. Caché des informations dans des images, dû au fait que l'oeil humain ne puisse pas déceler ces petites différences, peut s'apparenter à un domaine de sécurité plus vaste, qui s'appelle la Stéganographie. Une solution pour se prémunir de ce type d'attaque, serait d'entraîner le réseau sur des images volontairement 'floutés' par le défenseur, pour entraîner le système à être imperméable à ce genre d'attaque. Mais cela nécessite une reprise manuel de l'ensemble du jeu de données, et au vu du nombre de variations possibles, un jeu de données qui devient extrêment grand.
